# GitHub CDN 区域对比测试报告 — 香港 vs 韩国

## 测试概要

| 项目 | 内容 |
|---|---|
| 测试目标 | 对比 GitHub 香港 CDN 与韩国 CDN 节点的并发承受能力 |
| 测试方法 | 渐进式并发 git clone `--depth 1`，逐步缩短超时阈值 |
| 目标仓库 | `https://github.com/sgl-project/sglang.git`（大型仓库） |
| 测试日期 | 2026-07-10 |
| 香港 CDN IP | `20.205.243.166`（当前 DNS 默认解析） |
| 韩国 CDN IP | `20.200.245.247`（运维指定） |

## 分析与结论

### 1. 韩国节点低并发表现更优

在并发 ≤ 20 的区间，韩国节点表现出明显优势：
- 并发 5 时 clone 速度比香港快 **24%**（6.06s vs 7.99s）
- 并发 20 时 **零 504**，而香港已有 15% 的请求超时

这说明韩国 CDN 节点在**低-中等并发**区间有更大的吞吐余量。

### 2. 高并发区间两节点趋同

并发 ≥ 30 时，两个节点的 504 率接近（76% vs 83%），说明此时瓶颈已不在 CDN 节点本身，而在于：
- GitHub 后端服务处理能力上限
- 网络带宽/连接池饱和
- 代理超时阈值过低（10s 远不够大型仓库 clone）

### 3. 韩国节点的 504 馈线更晚

| 区域 | 504 馈线（首次出现 504 的并发量） |
|---|---|
| 香港 | **并发 20** |
| 韩国 | **并发 30** |

韩国节点能多承受 10 个并发请求才开始出现 504，相当于并发承受力 **提升 50%**。

### 4. 实际意义

如果当前服务部署在香港、通过香港 CDN 访问 GitHub：
- 并发 git clone/push 超过 **20** 就有 504 风险
- 切换到韩国 CDN 后，安全并发量提升到 **30**

建议在 CI/CD 管道中：
- 控制 git 操作并发 ≤ 当前区域馈线的 **70%**（香港 ≤ 14，韩国 ≤ 21）
- 配合 `--depth 1` 浅克隆和指数退避重试

## 建议

| 场景 | 建议 |
|---|---|
| CI/CD 管道在香港 | git 并发 ≤ 14，加 retry；或切换到韩国 CDN |
| CI/CD 管道可选区域 | 优先选择韩国节点（504 馈线更高） |
| 企业代理配置 | `proxy_read_timeout` ≥ 30s，`--depth 1` 浅克隆 |
| 高并发场景 | 分批执行 + 队列控制，避免同时触发 > 20 个 git 操作 |

## 测试脚本说明

```bash
python3 region_comparison.py
```
